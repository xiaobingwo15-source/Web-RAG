# Plan: Tenant Slug Login + Admin Approval for Signups

**Complexity:** 🔴 Complex — touches database, backend auth, frontend auth, admin UI

## Overview

Two tightly coupled features:
1. **Tenant slug in URL** — `/login?tenant=ie-industrial` links signup to the correct tenant
2. **Admin approval** — new signups are `pending` until admin approves them

## Data Flow

```
Client visits /login?tenant=ie-industrial
  → Frontend validates slug via GET /api/auth/tenant/{slug}
  → Client signs up → Supabase signUp with user_metadata: { tenant_slug }
  → DB trigger reads tenant_slug → resolves tenant_id → inserts profile with status='pending'
  → Client sees "Waiting for approval" screen
  → Admin sees new user in Users tab → clicks Approve
  → Profile status → 'approved' → client can now chat
```

## Implementation Steps

### 1. Database Migration (`021_signup_approval.sql`)

| Change | Detail |
|--------|--------|
| Add `status` column | `text NOT NULL DEFAULT 'approved'`, CHECK in `('pending','approved','suspended')` |
| Replace `handle_new_user()` trigger | Reads `tenant_slug` from `raw_user_meta_data`, resolves tenant_id, sets `status='pending'` |
| Add `is_approved_user()` function | Returns `true` if current user's profile status is `'approved'` |
| Update RLS on `threads` | INSERT/UPDATE/DELETE now also require `is_approved_user()` |
| Update RLS on `messages` | Same pattern — pending/suspended users blocked from writes |
| Add index | `idx_profiles_status` for fast admin queries |

Existing users unaffected — `DEFAULT 'approved'` keeps them working.

### 2. Backend — Auth Middleware (`auth.py`)

- Add `status` field to `AuthenticatedUser` class
- Fetch `status` from profiles in `get_current_user()`
- Return `status` in `GET /api/auth/me` response

### 3. Backend — Public Tenant Endpoint (`auth.py`)

- `GET /api/auth/tenant/{slug}` — no auth required
- Returns `{ id, name, slug }` or 404
- Used by frontend to validate slug before signup

### 4. Backend — Database Functions (`database.py`)

- `get_tenant_users(tenant_id)` — list all profiles for a tenant
- `update_user_status(tenant_id, user_id, new_status)` — approve/suspend
- Update `accept_tenant_admin_invite` to set `status='approved'`

### 5. Backend — Admin Endpoints (`admin.py`)

| Endpoint | Method | Action |
|----------|--------|--------|
| `/admin/users` | GET | List all users in admin's tenant |
| `/admin/users/{id}/approve` | POST | Set user status to `approved` |
| `/admin/users/{id}/suspend` | POST | Set user status to `suspended` |

### 6. Backend — Chat Guard (`chat.py`)

- Both `chat()` and `chat_stream()` check `user.status != "approved"` → 403

### 7. Frontend — useAuth Hook (`useAuth.ts`)

- Add `status` to auth context
- Update `signUp(email, password, tenantSlug)` — passes `tenant_slug` in `options.data`

### 8. Frontend — LoginPage (`LoginPage.tsx`)

- Read `?tenant=<slug>` from URL via `useSearchParams`
- Validate slug via `GET /api/auth/tenant/{slug}` on mount
- Show error if slug missing/invalid, disable form
- Pass `tenantSlug` to `signUp()`
- On login: verify user's tenant_id matches the slug's tenant

### 9. Frontend — ChatPage (`ChatPage.tsx`)

- `status === 'pending'` → "Account Pending Approval" screen with sign-out button
- `status === 'suspended'` → "Account Suspended" screen with sign-out button

### 10. Frontend — AdminPage Users Tab (`AdminPage.tsx`)

- New "Users" tab alongside existing "Chats" and "API Keys" tabs
- Table: Email, Role, Status, Joined, Actions
- Approve button for `pending` users, Suspend button for `approved` users
- Admin cannot suspend themselves

### 11. Frontend — API Functions (`api.ts`)

- Add `UserProfileResponse.status` field
- Add `getAdminUsers(token)` and `updateUserStatus(userId, action, token)`

## Verification

1. **Migration**: Run migration, verify existing users still have `status='approved'`
2. **Signup flow**: Visit `/login?tenant=ie-industrial`, sign up, verify profile has `status='pending'` and correct `tenant_id`
3. **Pending screen**: Log in as pending user, verify "Waiting for approval" screen appears
4. **Admin approval**: Log in as admin, go to Users tab, approve user, verify user can now chat
5. **Suspend**: Suspend user, verify they see "Account Suspended" screen
6. **RLS**: Verify pending user cannot create threads via direct API call
7. **Wrong tenant**: Login with `?tenant=wrong-slug`, verify error
