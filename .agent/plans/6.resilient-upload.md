# 6. Resilient Admin File Upload

## Problem

When admin uploads large files, three fragility points cause data loss on page refresh/navigation:

1. **Single HTTP POST** â€” entire file in one `fetch` call; browser aborts on navigation
2. **In-memory polling** â€” `useDocuments` hook unmounts = polling stops, no recovery
3. **Zero persistence** â€” nothing in localStorage; all state lost on any page lifecycle event

## Solution: Chunked Upload + Persistent Polling

Three layers, each independently useful:

| Layer | What | Survives refresh? |
|-------|------|--------------------|
| Chunked upload | File split into 2MB chunks, each a quick POST | âś… Lose at most 1 chunk |
| localStorage persistence | Session IDs + progress stored in localStorage | âś… State survives page reload |
| Polling recovery | On mount, check localStorage for pending docs, resume polling | âś… Processing status survives navigation |

### Step 1: Database Migration

CREATE TABLE public.upload_sessions (

### Step 2: Backend Upload Session Router

**New file:** `backend/app/routers/upload_session.py`

Five endpoints, all behind existing `get_current_user` + admin check:

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/documents/upload/init` | POST | Create session, return session_id + total_chunks |
| `/api/documents/upload/{id}/chunk/{index}` | POST | Receive one chunk (multipart), verify SHA-256 checksum |
| `/api/documents/upload/{id}/complete` | POST | Merge chunks â†’ create document â†’ trigger ingestion |
| `/api/documents/upload/{id}/status` | GET | Return session state (for resume after refresh) |
| `/api/documents/upload/{id}` | DELETE | Cancel session, clean up chunks |

**Completion flow** mirrors existing `documents.py` upload endpoint:
- Merge chunk files on disk â†’ read into memory â†’ compute content_hash â†’ duplicate check â†’ create document â†’ schedule `process_document_async` â†’ clean up chunk directory

**Chunk storage:** `uploaded_chunks/{session_id}/{chunk_index:06d}` on disk. Cleaned up after completion or cancellation.

**Stale session cleanup:** On startup, expire any `uploading` sessions older than 24 hours.

**Modify:** `backend/app/main.py` â€” register new router

### Step 4: Frontend Chunked Upload Hook

**New file:** `frontend/src/hooks/useChunkedUpload.ts`

Core hook replacing the single-file `apiUpload` call:

- **Chunking:** `File.slice(start, end)` into 2MB pieces
- **Progress:** `XMLHttpRequest` with `upload.onprogress` for per-chunk progress
- **Checksums:** SHA-256 via `crypto.subtle.digest` (SubtleCrypto)
- **Retry:** 3 attempts per chunk with exponential backoff
- **Persistence:** localStorage updated after each confirmed chunk
- **beforeunload:** Warning when active uploads exist
- **Resume:** On mount, check localStorage for incomplete sessions â†’ query server status â†’ show "re-select file to resume" banner

**Limitation:** `File` objects can't persist across page refreshes. On resume, user must re-select the same file. Hook matches by `filename + totalSize`.

### Step 5: Frontend API Functions

**Modify:** `frontend/src/lib/api.ts`

Add five functions: `initUploadSession`, `uploadChunk`, `completeUploadSession`, `getUploadSessionStatus`, `cancelUploadSession`.

`uploadChunk` uses `XMLHttpRequest` (not `fetch`) for progress events.

### Step 6: Polling Recovery

**Modify:** `frontend/src/hooks/useDocuments.ts`

On mount effect:
1. Fetch document list as usual
2. Load persisted polling document IDs from localStorage
3. For each ID with non-terminal status â†’ resume polling loop
4. On terminal status â†’ remove from localStorage

### Step 7: UI Updates

**Modify:** `frontend/src/components/DocumentUpload.tsx`

- Progress bar showing chunk progress (e.g., "Uploading report.pdf â€” 3/12 chunks, 25%")
- Resume banner: "Upload of [filename] incomplete. Re-select file to resume."
- Cancel button on in-progress uploads
- Use `useChunkedUpload` hook alongside existing `useDocuments`

### Step 8: Edge Case Hardening

- **SHA-256 checksum** on each chunk (frontend computes, backend verifies)
- **Stale session cleanup** on backend startup
- **Multi-tab awareness** via `BroadcastChannel` (optional, can defer)
- **Small file optimization:** Files < 2MB use existing single POST endpoint
