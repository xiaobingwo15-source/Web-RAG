# RAG Grounding Improvement Plan

**Date**: 2026-06-18
**Status**: ⬜ PENDING APPROVAL
**Complexity**: 🔴 Complex (multi-phase, cross-cutting changes)
**Goal**: Increase RAG answer grounding from current ~50-70% to 85%+

---

## Executive Summary

Four parallel research agents analyzed the RAG pipeline for grounding/faithfulness gaps. They found:

- **12 pipeline gaps** (4 HIGH, 6 MEDIUM, 2 LOW severity)
- **10 eval pipeline weak points** (no real baseline, primitive groundedness metric, golden test set missing)
- **10 researched improvement techniques** ranked by impact
- **6 advanced architecture patterns** (Self-RAG, CRAG, citation attribution, etc.)

The root cause: answers are not fully grounded because the groundedness checker is too permissive (token-overlap at 0.50 fast-path), uses the same model for self-verification, and web-fallback content bypasses document grounding checks entirely.

## Implementation Phases

### Phase 1: Quick Wins (1-2 days) ⚡
**Focus**: Fix the most impactful gaps with minimal code changes

#### 1.1 Pre-Generation Context Filtering
- **What**: Add hard score threshold after Cohere reranker — drop chunks below 0.3 relevance
- **Where**: `backend/app/services/retrieval.py` after `rerank_with_cohere()`
- **Why**: Prevents irrelevant context from reaching the LLM
- **Gap addressed**: M6 (partial)
- **Files**: `retrieval.py`

#### 1.2 Reranking Abstention
- **What**: When max reranker score < 0.25, return "no relevant documents found" instead of forcing generation on bad context
- **Where**: `backend/app/services/agents/doc_rag_agent.py` after reranking
- **Why**: Safety net — refuse rather than hallucinate
- **Gap addressed**: M5 (partial), H3 (partial)
- **Files**: `doc_rag_agent.py`

#### 1.3 Increase Groundedness Check Budgets
- **What**: Raise `c[:500]` → `c[:1500]` per chunk and `answer[:2000]` → full answer in LLM groundedness check
- **Where**: `backend/app/services/groundedness.py` lines 55-56
- **Why**: Current truncation causes the checker to miss ungrounded claims
- **Gap addressed**: H2
- **Files**: `groundedness.py`

#### 1.4 Remove Groundedness Fast-Path
- **What**: Remove the `GROUNDEDNESS_LLM_HIGH = 0.50` fast-path — always run LLM check
- **Where**: `backend/app/services/groundedness.py`
- **Why**: 50% token overlap is too permissive; paraphrased hallucinations pass
- **Gap addressed**: H1 (partial)
- **Files**: `groundedness.py`

#### 1.5 Web-vs-Document Groundedness Distinction
- **What**: When web fallback is used, tag web chunks in the groundedness check prompt and require the LLM to verify claims are supported by *document* chunks specifically
- **Where**: `backend/app/services/groundedness.py`, `doc_rag_agent.py`
- **Why**: Currently web-only answers pass groundedness check as "grounded"
- **Gap addressed**: H3
- **Files**: `groundedness.py`, `doc_rag_agent.py`

**Verification**: Run existing tests, manual test with a query that triggers web fallback, verify groundedness score drops for web-only answers.

### Phase 2: Claim-Level Grounding (3-5 days) 🎯
**Focus**: Replace coarse token-overlap with fine-grained claim verification

#### 2.1 Claim Decomposition
- **What**: After generating an answer, decompose it into atomic claims via LLM structured output
- **Where**: New function in `backend/app/services/groundedness.py`
- **Why**: Whole-answer checking misses partial hallucinations
- **Gap addressed**: H1 (full), GAP-10
- **Prompt**: "Decompose the following answer into atomic factual claims. Return a JSON array of strings."
- **Files**: `groundedness.py`

#### 2.2 Per-Claim Context Verification
- **What**: For each atomic claim, verify it is entailed by at least one retrieved chunk using LLM-as-judge
- **Where**: New function in `groundedness.py`
- **Why**: Catches paraphrased hallucinations that token overlap misses
- **Gap addressed**: H1, M1 (partial — uses structured output)
- **Files**: `groundedness.py`

#### 2.3 Sentence-Level Source Attribution
- **What**: Link each sentence in the answer to its best supporting chunk ID
- **Where**: New function, called after generation in `doc_rag_agent.py`
- **Why**: Enables per-sentence grounding scores and citation verification
- **Gap addressed**: GAP-10
- **Files**: `groundedness.py`, `doc_rag_agent.py`

#### 2.4 Update Groundedness Scoring
- **What**: Replace `check_groundedness()` with claim-level scoring: `groundedness = supported_claims / total_claims`
- **Where**: `groundedness.py`, update callers in `doc_rag_agent.py`
- **Why**: More accurate metric that catches partial hallucinations
- **Gap addressed**: H1
- **Files**: `groundedness.py`, `doc_rag_agent.py`

**Verification**: Run eval suite, compare groundedness scores before/after, verify partial hallucinations are caught.

---

### Phase 3: Chain-of-Verification (3-5 days) 🔗
**Focus**: Post-generation self-check to catch remaining hallucinations

#### 3.1 Verification Question Generation
- **What**: After generating a draft answer, generate fact-checking questions targeting each claim
- **Where**: New function in `backend/app/services/agents/doc_rag_agent.py`
- **Why**: Independent verification catches hallucinations the generator missed
- **Gap addressed**: M1 (same-model bias mitigated by independent questions)
- **Files**: `doc_rag_agent.py`

#### 3.2 Independent Verification
- **What**: Answer each verification question independently (separate LLM call, no draft answer context)
- **Where**: New function in `doc_rag_agent.py`
- **Why**: Prevents bias from seeing the draft answer
- **Files**: `doc_rag_agent.py`

#### 3.3 Answer Revision
- **What**: If verification reveals unsupported claims, revise the answer to remove or qualify them
- **Where**: Modify the generation flow in `doc_rag_agent.py`
- **Why**: Self-correction before the answer reaches the user
- **Files**: `doc_rag_agent.py`

**Verification**: Test with queries known to produce hallucinations, verify revision removes unsupported claims.

---

### Phase 4: Confidence Scoring & Routing (2-3 days) 📊
**Focus**: Multi-signal confidence to route queries through appropriate paths

#### 4.1 Confidence Score Fusion
- **What**: Combine retrieval scores, reranker scores, groundedness, and chunk count into a single confidence score
- **Where**: New function in `doc_rag_agent.py`
- **Why**: Existing signals are used in isolation; fusion provides better routing decisions
- **Files**: `doc_rag_agent.py`

#### 4.2 Confidence-Based Routing
- **What**: Route queries based on confidence:
  - HIGH (>0.8): Return with citations
  - MEDIUM (0.5-0.8): Re-retrieve with reformulated query
  - LOW (≤0.5): Uncertainty disclaimer + web fallback offer
- **Where**: Modify main generation flow in `doc_rag_agent.py`
- **Why**: Low-confidence answers should not be returned unchecked
- **Gap addressed**: L2 (replaces fixed retry count)
- **Files**: `doc_rag_agent.py`

#### 4.3 Adaptive Score Thresholds
- **What**: Replace hardcoded 0.4/0.5 thresholds with percentile-based grading relative to current retrieval batch
- **Where**: `doc_rag_agent.py` in `grade_retrieval_quality()`
- **Why**: Static thresholds don't adapt to different embedding models or score distributions
- **Gap addressed**: M5
- **Files**: `doc_rag_agent.py`

**Verification**: Test with queries of varying difficulty, verify routing decisions match expected confidence levels.

### Phase 5: Eval Pipeline Hardening (2-3 days) 🧪
**Focus**: Fix evaluation infrastructure to measure grounding improvements

#### 5.1 Create Golden Test Set

- **Where**: `backend/app/services/rag_eval.py` lines 81-83
- **Why**: Redundant metrics provide false confidence
- **Files**: `rag_eval.py`

#### 5.4 Run First Real Eval Baseline
- **What**: Execute eval suite and persist results as the real baseline (replacing placeholder 3.5)
- **Where**: `backend/tests/fixtures/eval_baseline.json`

- **Files**: `eval_baseline.json`

**Verification**: CI eval run completes without skipping, baseline persisted, threshold alignment verified.

### Phase 6: Structured Output + Citation Enforcement (3-5 days) 📝
**Focus**: Force machine-verifiable citations

#### 6.1 Structured Citation Output
- **What**: Switch LLM output to JSON mode with explicit `answer` + `citations` array
- **Where**: `backend/app/services/gemini.py` in `_build_messages()`
- **Why**: Prompt-only citation instructions are unreliable; structured output makes it machine-verifiable
- **Files**: `gemini.py`, `doc_rag_agent.py`

#### 6.2 Citation Verification
- **What**: Post-generation check that every `[N]` citation maps to a real chunk, and cited content exists in that chunk
- **Where**: New function in `groundedness.py`
- **Why**: Catches hallucinated citations
- **Files**: `groundedness.py`

#### 6.3 HyDE Relevance Backcheck
- **What**: After HyDE retrieval, verify chunks are relevant to the *original* query (not just the hypothetical answer)
- **Where**: `doc_rag_agent.py` after HyDE retrieval
- **Gap addressed**: M3
- **Files**: `doc_rag_agent.py`

**Verification**: Test with multi-source queries, verify all citations map to real chunks, HyDE backcheck rejects off-topic chunks.

## Dependencies & Risks

| Risk | Mitigation |
|------|------------|
| Claim decomposition adds latency (~2s per answer) | Make it optional via env var, enable for high-stakes queries only |
| CoVe adds 2 extra LLM calls per answer | Use smaller model for verification, batch verification questions |
| Structured output may break existing streaming | Implement as optional mode, fallback to free-text |
| Removing groundedness fast-path increases LLM costs | Acceptable — grounding accuracy > cost savings |
| Golden test set creation requires manual validation | Use LLM-generated set with admin review gate |
